# README 评估逻辑说明设计

## 目标

在中文 `README.md` 中补充与当前实现一致的评估逻辑、评估维度和项目特色，让首次接触项目的用户无需阅读 Python 源码，也能理解报告如何形成研究信号、哪些数据参与判断，以及结果有哪些边界。

## 内容位置

在“支持标的”之后、“快速开始”之前新增以下内容：

1. `核心评估逻辑`：说明报告采用两层评估，而不是把所有指标直接相加。
2. `评估维度`：列出最终研究信号的组成、数据来源和解释方式。
3. `存储周期内部评分`：公开六个存储专属模块及公司差异化规则。
4. `研究信号区间`：解释分数只代表当前数据下的研究强弱，不是交易评级。
5. `Skill 特色`：总结一手来源刷新、事实与估算分离、公司差异化、安全登录、可追溯报告和历史样本诊断。

## 评估逻辑口径

### 第一层：存储周期评分

存储周期评分从 50 分起步，按库存周期、DRAM/NAND 价格周期、CapEx、HBM 供需、下游需求和技术节点进行加减分，最后限制到 `0–100`：

```text
C = clamp(50 + Δ库存 + Δ价格 + ΔCapEx + ΔHBM + Δ下游 + Δ技术节点, 0, 100)
```

其中 HBM 与下游需求按公司业务结构差异化：HBM 主要供应商、部分涉及公司和不涉及公司使用不同暴露系数；下游需求按 DRAM、NAND、HBM、HDD 的公司收入结构加权。README 使用“评分贡献”描述这些规则，不把所有加减分误写成严格归一化百分比。

### 第二层：最终研究信号

最终研究信号使用下列实现公式：

```text
S = clamp(
    0.10×短期技术
  + 0.25×存储周期
  + 0.12×分析师
  + 0.08×HBM供需
  + 0.08×下游需求
  + 0.08×内部人
  + 0.08×股东变动
  + 0.04×技术节点
  + 0.05×行业对标
  + 财务质量修正,
  10, 95
)
```

前九项是评分系数，财务质量修正是净利率、毛利率、ROE、负债率、PB 和 Beta 的直接加减分，不将其描述为百分比权重。README 不宣称这些系数构成归一化到 100% 的资产定价模型。

### 分数区间

按最终报告当前逻辑解释：

| 最终研究信号 | 解释 |
| --- | --- |
| `65–95` | 研究指标整体较强，仍需复核数据新鲜度和模型假设 |
| `50–64` | 指标处于中间区间，重点核验周期拐点和缺失数据 |
| `35–49` | 多项指标偏弱，需要继续核验基本面和数据时点 |
| `10–34` | 风险或数据质量信号较多，优先检查数据完整性 |

同时明确：分数不是买卖建议、目标价、上涨概率或收益承诺。

## 评估维度

README 以表格说明以下维度：

- 短期技术：RSI、MACD Histogram、量比；
- 存储周期：库存、价格、CapEx、HBM、下游需求、技术节点；
- 分析师共识；
- HBM 供需模型；
- 公司加权下游需求；
- 内部人交易；
- 股东持仓变动；
- 公司技术节点；
- 同行 PE 对标；
- 财务质量修正；
- 数据质量与历史样本诊断，作为解释和验证层，不包装成未来收益证明。

## 项目特色

README 突出以下可验证特性：

1. 每次分析先刷新 GPU/HBM 五个必需模块，并通过一次性运行清单门禁。
2. 将官方事实、第三方资料和模型估算分开显示，估算保留公式、假设、置信度和局限。
3. 按公司业务结构调整 HBM、下游需求和技术节点影响，避免把 MU 的逻辑直接套用到 WDC/STX。
4. DRAM/NAND 行业价格与公司毛利率交叉验证，降低单一来源误判。
5. 报告展示来源、`as_of`、`verified_at`、数据新鲜度和旧快照授权状态。
6. Windows 与 macOS 使用本机可见临时终端登录，密码字符隐藏且凭据不进入聊天、报告或日志。
7. 交互式 HTML 同时提供评分拆解和历史样本诊断；回测只用于检查历史相关性。
